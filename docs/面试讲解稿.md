# 面试讲解稿

## 项目名称
RuoYi v3.9.2 后台管理系统接口与 UI 自动化测试项目

## 简历描述
基于 RuoYi v3.9.2 原版前后端分离后台管理系统，使用 Python + pytest + requests + Playwright 搭建综合自动化测试框架，覆盖登录认证、字典管理、部门管理、岗位管理、用户管理、角色管理、菜单权限等核心模块，实现接口自动化、UI 自动化、接口联动、数据库校验、失败截图、Trace 回放、Allure 报告和 Jenkinsfile 流水线配置。

## 技术栈
Python、pytest、requests、Playwright、Page Object、YAML、pymysql、Allure、Jenkinsfile、Git、JMeter

## 6 大亮点
1. **接口与 UI 分层覆盖**：接口验证业务逻辑、异常和数据库，UI 验证真实页面操作。
2. **接口联动 + 数据库校验**：接口造数、接口查询、数据库校验、接口清理形成闭环。
3. **UI 自动化采用 Page Object 分层**：页面元素和动作封装进 pages 层，用例只描述流程和断言。
4. **权限场景设计**：通过角色 `menuIds` 和 `sys_role_menu` 验证菜单权限关系。
5. **失败定位能力**：UI 失败自动截图，并保留 Playwright Trace 文件。
6. **工程化**：支持环境变量配置、日志/Allure 脱敏、Jenkinsfile 报告生成配置。

## 9 问参考答案

### Q1 为什么选择 RuoYi？
因为 RuoYi v3.9.2 是完整的前后端分离后台管理系统，包含登录、字典、部门、岗位、用户、角色、菜单权限等典型后台模块，适合展示接口自动化、UI 自动化、权限测试和数据库校验能力。

### Q2 接口自动化框架怎么设计？
我将接口自动化分为 base、clients、testcases、common 四层。BaseApi 封装 get/post/put/delete、headers、token、日志和 Allure 附件；clients 按业务模块封装接口方法；testcases 只写测试流程和断言；common 放配置、日志、随机数据、数据库、断言、脱敏和清理工具。

### Q3 UI 自动化框架怎么设计？
UI 采用 Page Object 模型，分 base、pages、testcases、common 四层。BasePage 封装打开页面、等待、表格行、弹窗、输入、截图等公共能力；pages 封装页面动作；testcases 只写业务流程。

### Q4 接口和 UI 怎么配合？
接口更适合造数、校验业务逻辑和清理；UI 负责验证真实页面操作。比如字典、部门、岗位等场景先通过接口准备数据，再用 UI 查询或操作，最后用接口和数据库验证结果。

### Q5 怎么处理 token？
登录接口 `/login` 返回 `token` 字段，BaseApi 统一把它放入 `Authorization: Bearer <token>` 请求头。logout 用例使用独立 token，避免污染 session 级管理员 token。

### Q6 怎么处理测试数据？
所有自动化数据都使用本轮运行唯一前缀 `TEST_RUN_PREFIX`，避免并发运行互相清理。测试结束优先接口清理，必要时数据库兜底，但清理范围只限制在本轮前缀数据内。

### Q7 为什么要做数据库校验？
接口返回成功不代表数据一定正确落库。对增删改和权限关系类场景，我会查 `sys_user`、`sys_role`、`sys_role_menu` 等表验证字段和关系是否符合预期。

### Q8 UI 自动化用例不稳定怎么办？
优先使用 Playwright 的 `get_by_role/get_by_text/get_by_placeholder` 语义化定位；避免 sleep；失败时自动截图和保留 Trace；对于 element-ui 弹窗，使用 `role=dialog` 定位。

### Q9 JMeter 性能测试怎么做？
我编写了 JMeter 性能测试方案和可导入的 `.jmx`，场景是登录提取 token，再携带 `Authorization` 查询字典分页接口。当前未执行真实压测，所以简历只写“设计/编写 JMeter 性能测试方案”，不写“完成性能测试并达标”。
