# 测试计划

## 1. 测试目标

验证 **RuoYi v3.9.2 原版后台管理系统** 的核心模块接口、UI 页面和接口联动/数据库校验流程是否符合预期，并形成可复用的自动化测试框架。

## 2. 测试范围

- 登录认证
- 字典管理
- 部门管理
- 岗位管理
- 用户管理
- 角色管理
- 菜单权限
- UI 自动化
- 接口联动与数据库校验
- JMeter 性能测试方案设计（未执行真实压测）

## 3. 非测试范围

- 支付、工作流、商城订单、复杂多租户、大屏报表、安全深度测试。
- 当前未产出 JMeter `.jtl` 和 HTML 性能报告，因此不声明“完成性能压测”。

## 4. 测试环境

| 项 | 值 |
|---|---|
| 后端 | `http://localhost:8080` |
| 前端 | `http://localhost:8081` |
| 数据库 | Docker `ruoyi-mysql`，端口 `13306`，库 `ry-vue` |
| Redis | Docker `ruoyi-redis` |
| 账号 | `admin/admin123` |
| 验证码 | `sys_config.sys.account.captchaEnabled=false` |

## 5. 进入准则

- MySQL、Redis、后端、前端均可访问；
- 管理员账号可登录；
- 数据库账号可查询测试表；
- Playwright 浏览器已安装；
- `.venv` 依赖安装完成且 `pip check` 无冲突。

## 6. 退出准则

- 152 个测试实例执行完成；
- Allure 报告生成；
- 自动化创建的本轮测试数据已清理；
- 失败用例有明确原因（当前最终结果为 152 passed）。

## 7. 风险与应对

| 风险 | 应对 |
|---|---|
| 验证码导致登录失败 | 数据库关闭 `sys.account.captchaEnabled` |
| UI 定位随前端版本变化 | 使用 `get_by_role/get_by_text/get_by_placeholder`，并保留 Trace |
| 测试数据残留 | 使用运行级唯一前缀 `TEST_RUN_PREFIX` + 精确清理 |
| token 被 logout 污染 | logout 使用 function 级独立 token |
| 日志/Allure 泄露敏感信息 | BaseApi 输出前统一脱敏 |
| 并发测试误清理其它数据 | 清理仅限定本轮 `TEST_RUN_PREFIX%` |
