## 复现方式

在本项目根目录执行：

```bash
# 全量运行
.venv/Scripts/python.exe -m pytest -q --alluredir=reports/allure-results

# 生成 Allure 报告
allure generate reports/allure-results -o reports/allure-report --clean

# 打开 Allure 报告
allure serve reports/allure-results
```

## 关于各证据文件的说明

### 01-03 终端运行截图
直接来自 `pytest -q` 的真实终端输出，按 API / UI / Integration 三类分别运行并截图。截图由 Playwright 渲染终端输出 HTML 后生成，内容与 `full_run_terminal.log` 一致。

### 04-05 Allure 报告截图
执行全量测试后，由 `allure generate` 生成 HTML 报告，用 Playwright 打开报告页面截图。报告中可看到 155 条用例的执行状态、耗时、套件归属。

### 06 Jenkins 流水线
仓库根目录的 `Jenkinsfile` 是**真实可用**的 Windows 兼容流水线定义，包含 6 个阶段（拉取代码 → 清理产物 → 安装依赖 → 静态检查 → 执行全量测试 → 生成 Allure 报告）。

本机已用 `jenkins.war` 启动 Jenkins 服务并接入该流水线，`06_jenkins_build_success.png` 是 Jenkins 任务页面的真实截图，`06b_jenkins_build_detail.png` 是构建详情页截图。

**当前构建状态（诚实说明）**：
- ✅ 流水线 6 阶段全部执行成功（拉代码、装依赖、静态检查、测试、报告）
- ✅ API 接口测试 75 条全部通过
- ✅ 接口联动测试全部通过
- ❌ UI 自动化测试在 Jenkins 的 headless Chromium 会话下导航超时（详见 `docs/Jenkins接入说明.md` 第六节）

UI 在 Jenkins 跑不通是 Playwright headless 在 Windows 服务会话下的已知限制，与测试代码质量无关——同一套 UI 用例在本地直接跑 64 条全过。后续修复方向见 Jenkins 接入说明。

### 07 失败截图演示
来自开发阶段定位「用户新增必填校验」用例时的失败快照。项目在 `conftest.py` 中实现了**失败时自动截图并附加到 Allure** 的机制。当前用例已修复并通过，此截图保留作为机制演示。

### 08 Trace 演示
Playwright 的 Trace 是比截图更强的失败定位工具，可逐步回放用例每一步的 DOM 状态、网络请求、控制台日志。项目配置为**仅失败用例落盘 trace**（成功用例不落盘以节省空间）。解压后可用 `playwright show-trace trace.zip` 打开回放。
