2. Apifox / Postman 是否只学一个
结论：可以只深学一个。对你当前求职阶段，建议“Apifox 深学 + Postman 中等掌握”。
工具
学习深度
为什么这样分配
Apifox
主力深练
适合国内项目团队，接口文档、调试、Mock、自动化场景一体化；和 ruoyi-vue-pro 的 Swagger / OpenAPI 导入场景匹配度高。
Postman
会迁移、会 Runner、会 Newman
面试常见，公司也常用；重点不是重复深学，而是能把环境变量、token、断言、集合运行迁移过去。
两者共同点
必须理解
环境变量、接口集合、Header、前置/后置脚本、断言、批量运行、报告输出。
面试表达
推荐说法
“我主要用 Apifox 管理接口和场景化调试，同时掌握 Postman 的 Collection、Tests 断言、Runner 和 Newman，所以公司使用 Postman 也能快速迁移。”
2.1 不建议的学习方式
不建议 Apifox 和 Postman 都从零深挖高级功能，容易工具堆砌，浪费时间。
不建议只会点“发送请求”，不会环境变量、token 提取、断言和批量运行。
不建议简历写“熟练 Postman/Apifox/JMeter/Jenkins/Fiddler”，但没有截图、报告、脚本、项目仓库支撑。
3. ruoyi-vue-pro 项目准备与接口练习范围
训练时以本地启动的 ruoyi-vue-pro 后端作为被测系统。若你的本地端口、接口路径、默认账号与文档不一致，一律以你本地 Swagger / Knife4j / Apifox 导入后的实际接口为准。

12.2 面试高频问答
问题
回答要点
Apifox 和 Postman 你怎么取舍？
主用 Apifox 做接口文档、调试、环境、断言、场景；Postman 掌握 Collection、Tests、Runner、Newman，能快速迁移。
接口请求前端能成功，工具里失败怎么排查？
先用 Fiddler 抓前端真实请求，对比 baseUrl、Method、Header、tenant-id、Authorization、Body、Query 参数。
token 怎么处理？
登录接口后置脚本提取 accessToken 到环境变量，后续接口统一 Header：Authorization: Bearer {{accessToken}}。
JMeter 压测为什么不用 GUI？
GUI 适合调试脚本，正式负载测试用非 GUI 模式减少客户端资源影响，并生成 jtl/HTML 报告。
性能报告看哪些指标？
Average、90/95/99 分位、TPS、Error %、Samples，并结合后端日志分析瓶颈。
Jenkins 在项目里做了什么？
拉取测试仓库、安装依赖、执行 pytest/Newman/JMeter、归档报告，失败时通过控制台和报告定位。
Fiddler 有什么用？
抓真实前端请求，定位 Header/参数/响应差异；能重放请求、修改参数、导出 SAZ 给开发复现。
你如何保证练习不是工具堆砌？
每个工具都绑定 ruoyi-vue-pro 的真实接口和产出物：接口集合、压测报告、Jenkinsfile、抓包分析、README。
